Task General Description

You will build a full-stack agentic internal knowledge chatbot for Atlas Industries, a fictional company. An employee chats with the assistant through a web UI and asks things like "How do I claim a taxi from the airport on a business trip?", "What's the policy on remote work for new hires?", or "My laptop won't connect to the VPN, what do I do?".

The Workflow. Behind the chat, a LangGraph agentic workflow handles each question. At minimum the graph should: route the question into one of three domains (HR, IT Support, Finance & Reimbursement); retrieve relevant context from the right slice of the corpus; decide whether a tool call would help; maintain session memory so the user can ask follow-up questions naturally; and answer with a short, cited response. The exact node layout is your design decision — document it in the final report.

The Corpus. The corpus is intentionally messy. You will ingest 30 documents across HR, IT, and Finance — in .md, .pdf, and .docx formats, in English, Arabic, and bilingual variants. Your loader, splitter, and embedding choices must work cross-format and cross-language. This is part of the grade.

The Answer. Every answer must show its sources: the document filename (and ideally the snippet) that supports each claim. If the answer is not in the corpus, the assistant says so clearly and never invents one. Every run must be observable: a single run_id follows the question through routing, retrieval, tool calls, and the final answer, written to a structured log.

The Evaluation. A DeepEval evaluation script measures, on a fixed set of 10 gold questions, the following: routing accuracy (custom check — did the router pick the right domain?), and the standard DeepEval RAG metrics — Faithfulness, Answer Relevancy, Contextual Precision, Contextual Recall, and Hallucination.

The Scope. You are not building a fine-tuned model, a full enterprise SSO integration, or a production-grade multi-tenant system. You are building a focused, high-quality agentic RAG pipeline with clear observability and honest evaluation — the kind of project an examiner can grade in 20 minutes and a recruiter can demo in 5.

Suggested Technologies (Not Mandatory)

The only required pieces are: Python, LangGraph for the agentic workflow, an LLM, and DeepEval for evaluation. Everything else is your call — pick what fits, justify it in the final report.

Common options you might consider:

  • LLM & embeddings: Google Gemini, OpenAI, Groq, or a local Ollama model — whichever you can use for free. Since the corpus is mixed English / Arabic, pick embeddings that handle both languages well (e.g. BAAI/bge-m3, intfloat/multilingual-e5-large, or Cohere multilingual).
  • RAG primitives: LangChain, LlamaIndex, or your own loader / splitter / retriever code.
  • Document loaders: the corpus has .md, .pdf, and .docx. Consider pypdf or pdfplumber for PDFs, python-docx or unstructured for Word, and a plain reader for Markdown.
  • Vector store: FAISS, Chroma, Qdrant, Weaviate, or pgvector — local or free-tier cloud.
  • Chat front-end: Chainlit, Streamlit, Gradio, or a minimal FastAPI + simple HTML UI.
  • Session memory: LangGraph's built-in MemorySaver / checkpointer, LangChain memory classes, or your own in-process store keyed by session ID.
  • Tool schemas / validation: Pydantic is the conventional choice.
  • Logging: Python logging with a JSON formatter, structlog, or loguru.
  • Evaluation (required): DeepEval — FaithfulnessMetric, AnswerRelevancyMetric, ContextualPrecisionMetric, ContextualRecallMetric, and HallucinationMetric, plus a small custom routing-accuracy check. Make sure your judge LLM handles Arabic well.
  • Secrets: python-dotenv or any equivalent — never commit API keys.

In your final report, write one short paragraph per major choice (LLM, vector store, chat UI, eval approach) explaining why you picked it over the alternatives. This is what interviewers actually ask about.

Real Case Scenario

Atlas Industries is a fictional mid-sized company with around 500 employees. Today, when an employee asks "How do I get reimbursed for a client dinner?" they either ping a colleague, dig through an outdated intranet, or open a ticket and wait two days. The HR, IT, and Finance teams answer the same dozen questions over and over.

Anchor your demo in this run:

  • Turn 1: "I'm a new joiner. I traveled to a client meeting last week and paid for a taxi and a hotel. How do I submit the reimbursement, and how many days do I have to do it?" → Router picks Finance, retrieval over FIN-001-travel-reimbursement-policy.md and FIN-002-expense-claim-sop.pdf, one tool call to look up the submission window, cited answer.
  • Turn 2 (follow-up — exercises memory): "And what's the cap on the hotel for a domestic trip?" → The agent uses session memory to know "domestic trip" refers to the same scenario, retrieves the cap (1,500 EGP per night), cites the same policy.
  • Turn 3 (Arabic — exercises cross-language retrieval): "كم الحد الأقصى لعشاء العملاء لكل شخص؟" → Router picks Finance, retrieves FIN-009-عشاء-العملاء.docx, returns "2,000 جنيه مصري لكل شخص" with the source.

This three-turn scenario is your demo target. Whatever you build, it should handle these turns cleanly and produce defensible cited answers in front of your supervisor.

Dataset

The dataset is provided to you as a single bundle, atlas-corpus.zip. It contains the full document corpus and the gold evaluation cases.

Contents:

  • data/docs/hr/ — 10 HR documents (vacation, sick leave, onboarding, remote work, code of conduct, performance review, leave types, hiring, flexible hours, grievance).
  • data/docs/it/ — 10 IT Support documents (VPN, password reset, laptop provisioning, software request, incident response, acceptable use, email, network, MFA).
  • data/docs/finance/ — 10 Finance documents (travel reimbursement, expense claim SOP, procurement, per diem, corporate card, invoice approval, international travel, transport allowance, client dining, vendor payment).
  • tests/eval_cases.jsonl — 10 gold evaluation cases.

Why This Corpus Is Hard

Three file formats. Each domain has roughly 4 Markdown, 3 PDF, and 3 DOCX files. Your loader must handle all three — text extraction quirks differ between them (PDFs can break paragraphs mid-sentence, DOCX preserves structure better, Markdown is clean but headings need stripping).

Three language profiles. Roughly 60% of documents are English, 30% are Arabic, and 10% are bilingual (EN + AR side by side). Picking the wrong embedding model means an Arabic question fails to retrieve an Arabic document, even when the answer is right there. Use multilingual embeddings.

Right-to-left filenames. Some filenames contain Arabic characters (e.g. FIN-009-عشاء-العملاء.docx). Your ingestion pipeline and your citation logic must preserve these correctly through reading, indexing, and rendering.

Cross-document questions. Several evaluation questions need facts from two different documents (for example, the question about hotel caps + submission window requires both FIN-001 and FIN-002). Single-document retrieval will lose points on Contextual Recall.

Each line in tests/eval_cases.jsonl is a JSON object with the following fields:

Setup Steps

  1. Unzip atlas-corpus.zip at the root of your repo. The provided folder structure (data/docs/{hr,it,finance}/ and tests/eval_cases.jsonl) must be preserved exactly. Do not rename files — the eval cases reference them by filename.
  2. Open the inner README.md to skim the locked facts before writing your ingestion code.
  3. Verify your repo has all 30 documents and that filenames containing Arabic characters open correctly on your OS.
  4. Confirm your chosen PDF and DOCX loaders extract the Arabic text correctly. If your loader returns scrambled glyphs for the Arabic PDFs, switch tools — this is a known sign of bad extraction.

Known concerns to acknowledge: the corpus is small (30 documents), so retrieval ranking is sensitive to chunk size. Document this choice. Also, the bilingual documents will appear in retrieval results for both languages — that's fine, just make sure your prompt asks the model to respond in the user's language.

Functional Requirements

A. Environment & Setup

  • FR-A1 — Reproducible Python environment. Create a Python 3.10+ virtual environment with pinned versions in requirements.txt. A fresh clone + pip install must succeed without manual edits.
  • FR-A2 — Project structure. Implement the directory layout in Final Deliverables. Top-level packages contain an __init__.py where imports require it.
  • FR-A3 — Secrets in .env. API keys and tokens are read from environment variables via python-dotenv (or equivalent). Commit only .env.example.
  • FR-A4 — Configuration file. Centralize tunable values (paths, top-k, chunk size, model names, max-step limit) in src/settings.py. Hard-coded values scattered through the codebase fail this requirement.
  • FR-A5 — Git hygiene. Include a .gitignore that excludes .venv/, __pycache__/, the vector index, and .env.

B. Corpus & Ingestion

  • FR-B1 — Place the corpus as provided. Unzip into data/docs/hr/, data/docs/it/, data/docs/finance/. Do not rename, merge, or remove files.
  • FR-B2 — Multi-format loader. Implement loaders for .md, .pdf, and .docx. The ingestion script must not silently skip a format it doesn't recognize — it must fail loudly.
  • FR-B3 — Arabic-safe extraction. Verify Arabic and bilingual documents are extracted with correct character order (RTL preserved, no scrambled glyphs). Sample the output of at least one Arabic PDF and one Arabic DOCX in your EDA notebook.
  • FR-B4 — Chunking. Pick a chunking strategy (recursive character splitter, semantic splitter, or paragraph-based) and document the choice. Chunk size and overlap live in settings.py.
  • FR-B5 — Metadata tagging. Every chunk carries at minimum: domain (hr/it/finance), source_filename, and language (en/ar/bilingual). Citations depend on this.
  • FR-B6 — Multilingual embeddings. Use an embedding model that handles English and Arabic. Justify the choice in the final report.
  • FR-B7 — Index persistence. The vector index is built once and persisted to disk. A re-run of the app loads the existing index unless explicitly told to rebuild.
  • FR-B8 — Reset script. Provide scripts/reset.sh (or scripts/reset.py) that wipes the index and rebuilds it from data/docs/ in one command.

C. Retrieval

  • FR-C1 — Domain-filtered retrieval. The retriever accepts a domain (hr / it / finance) and filters chunks by the metadata tag. Bilingual documents must be retrievable regardless of the question's language.
  • FR-C2 — Top-k configurable. The number of retrieved chunks (top-3, top-5, etc.) is a config value, not a hard-coded constant.
  • FR-C3 — Source preservation. Retrieved chunks include the source filename in the returned object. The downstream answer step uses this for citations.
  • FR-C4 — Cross-language retrieval works. An Arabic question must surface relevant Arabic chunks even if the corpus contains English near-matches. Verify in notebooks/02_retrieval.ipynb with sample queries from eval_cases.jsonl.
  • FR-C5 — Optional reranking. If you add a reranker (cross-encoder, LLM-judged rerank, MMR), document it. Reranking is not required.

D. LangGraph Workflow

  • FR-D1 — Required responsibilities. The graph includes at least: a router node, a memory loader, a retriever node, an agent node that decides on tool calls, an answer node that composes a cited reply, and a memory writer. The node layout itself is your design choice.
  • FR-D2 — Router with fallback. The router classifies the question into hr / it / finance. When confidence is low, a sensible fallback (e.g. fan out to all domains, ask the user, or default-to-most-likely) is taken. Document the choice.
  • FR-D3 — State schema. The graph state is a typed object (Pydantic / TypedDict). At minimum it carries: question, domain, retrieved_chunks, tool_calls, answer, sources, session_id, run_id.
  • FR-D4 — Max-step counter. Enforce a per-run maximum step count (recommended 8 to 12). On overflow, route to a fallback node that answers with explicit uncertainty rather than looping forever.
  • FR-D5 — Max tool-call limit. Enforce a per-run maximum tool-call count (recommended 3). On overflow, the agent must answer with whatever it has.

E. Session Memory

  • FR-E1 — Full-session memory. Within a chat session, every prior turn (user question + agent answer) is available to the agent. A follow-up like "and what about international trips?" must resolve correctly against the previous turn.
  • FR-E2 — Session scoping. Memory is keyed by session_id (or LangGraph thread ID). A new session has empty memory.
  • FR-E3 — Reset on new session. Starting a fresh chat clears memory. Cross-session persistence is optional (bonus).
  • FR-E4 — Demoable in the recording. The 60–90 second demo must include at least one multi-turn exchange where memory clearly matters.

F. Tools

  • FR-F1 — Three typed tools, minimum. Each tool has a Pydantic input schema and a typed return. Suggested ideas: a policy lookup by ID (e.g. FIN-VAC-001), a reimbursement calculator over a list of items, a list-leave-types tool.
  • FR-F2 — Validation errors recover. Bad tool input is caught by Pydantic, returned to the agent as a feedback message, and the agent gets one more attempt. The app does not crash.
  • FR-F3 — Tool selection is sensible. The agent calls a tool only when it actually helps — not on every turn. Document the heuristic.

G. Answers & Citations

  • FR-G1 — Every answer is cited. Each answer ends with a short "Sources" section listing the document filenames used. If the answer is not in the corpus, the answer reads "I couldn't find this in the company documents" — never invent one.
  • FR-G2 — Source list is real. The cited filenames are filenames that actually exist in data/docs/. Invented filenames are an immediate fail.
  • FR-G3 — Answer in the user's language. If the user asks in Arabic, the answer is in Arabic. If the user asks in English, the answer is in English. The prompt should make this explicit.
  • FR-G4 — No fabricated policies. The agent must not invent a policy ID, an amount, a date, or a procedure that is not in retrieval. This rule is enforced by the DeepEval Faithfulness and Hallucination metrics.

H. Chat UI

  • FR-H1 — Single-command launch. The chat UI launches with one command (e.g. python app.py or chainlit run app.py). Framework is your choice.
  • FR-H2 — Workflow steps visible. The UI displays, for each turn: the chosen domain, the retrieved source filenames, any tool calls, and the final cited answer. Collapsible panels are fine.
  • FR-H3 — Right-to-left rendering. Arabic answers render correctly (RTL direction). Test with one of the Arabic eval questions before submission.
  • FR-H4 — New-session control. The UI has a clearly labeled "New chat" or "Reset" control that starts a fresh session.

I. Logging & Observability

  • FR-I1 — Run identifier. Each question is assigned a unique run_id (UUID) at the start of the graph. The same run_id tags every event for that turn.
  • FR-I2 — Structured log line per event. One JSON line per event — router decision, retrieval hits (with filenames), each tool call (with input and output), final answer — written to outputs/run_logs.jsonl.
  • FR-I3 — Demoable inspection. A grader should be able to grep by run_id and read the full timeline of one question end to end.
  • FR-I4 — No PII in logs. Logs may contain the question and answer, but should not contain raw API keys or full document bodies. Use chunk IDs / filenames, not entire chunks.

J. Evaluation (DeepEval)

  • FR-J1 — Evaluation script. tests/evaluate.py reads tests/eval_cases.jsonl, runs each case through the workflow, and produces outputs/eval_report.json.
  • FR-J2 — Test-case construction. Each case is converted to a DeepEval LLMTestCase with input, actual_output, expected_output, retrieval_context, and context.
  • FR-J3 — Required metrics. Report all five: FaithfulnessMetric, AnswerRelevancyMetric, ContextualPrecisionMetric, ContextualRecallMetric, HallucinationMetric.
  • FR-J4 — Custom routing accuracy. Compare each case's chosen domain against expected_domain. Report overall accuracy and per-case pass/fail in the same JSON report.
  • FR-J5 — Judge LLM choice. The judge LLM must handle Arabic. If your primary LLM is weak on Arabic, use a different one as the judge — document the choice.
  • FR-J6 — Report structure. eval_report.json includes per-metric averages, pass/fail counts, and per-case detail (id, input, predicted domain, expected domain, actual output, expected output, metric scores).

K. Documentation & Demo Assets

  • FR-K1 — README structure. Project overview, screenshots, 5-command Quick Start, results table, architecture diagram, file tree, known limitations.
  • FR-K2 — Architecture diagram. Produce docs/architecture.png showing the LangGraph node layout and the end-to-end flow.
  • FR-K3 — Screen recording. Record a 60–90 second walkthrough as docs/demo.gif or demo.mp4, including at least one multi-turn exchange that exercises memory and at least one Arabic question.
  • FR-K4 — Final report. A 3-page outputs/final_report.md covering the architecture, the stack choices (and why), the LangGraph design (with the memory mechanism), the RAG design, the routing logic, the tools, the DeepEval results (with a short comment on each metric), and known limitations.

Non-Functional Requirements

Non-functional requirements describe how well the system behaves — cost, performance, code quality, security, reliability. A project that passes the functional checklist but fails these will still feel amateur to an examiner.

A. Cost & Licensing

  • NFR-A1 — Zero-dollar build. Every dependency, model, and tool runs on a free tier or locally. No credit card, no paid cloud account, no paid embedding API.
  • NFR-A2 — License compatibility. Every Python package is under a permissive license (MIT / BSD / Apache 2.0). Add a LICENSE file to the repo (MIT is fine).
  • NFR-A3 — No real employer data. The corpus is 100% fictional Atlas Industries content. Do not add, replace, or augment with any document from a real employer.

B. Performance

  • NFR-B1 — End-to-end latency. A typical question (after the graph and embeddings are warm) returns an answer in under 8 seconds on a normal laptop. Cold-start latency is allowed to be higher.
  • NFR-B2 — Ingestion time. Full ingestion of the 30-document corpus completes in under 2 minutes on a normal laptop, no GPU.
  • NFR-B3 — UI responsiveness. The chat UI shows incremental progress (e.g. "Routing…", "Retrieving…") rather than freezing while the graph runs.
  • NFR-B4 — Memory footprint. The running app stays under 1 GB of RAM. Do not accidentally pickle the entire raw corpus inside the index.

C. Code Quality

  • NFR-C1 — One responsibility per file. No file in src/ exceeds about 250 lines. A reviewer can navigate the repo in 5 minutes.
  • NFR-C2 — Type hints. Every public function in src/ has typed parameters and a typed return.
  • NFR-C3 — Docstrings. Every public function has a summary, args, and returns docstring. Non-obvious lines (router prompt, retrieval reranking, threshold values) have a one-line comment explaining why.
  • NFR-C4 — Formatting. Run black on every Python file before committing. Mention this in the README.
  • NFR-C5 — No dead code. Remove commented-out experiments, unused imports, abandoned functions, and TODO markers before submission.
  • NFR-C6 — Linter clean. ruff check . produces zero errors. Warnings are OK if justified in the README.

D. Reliability & Robustness

  • NFR-D1 — Bad tool input recovers. A tool call with invalid input is caught and surfaced to the agent as feedback; the agent has one more chance to answer.
  • NFR-D2 — Empty retrieval handled. When retrieval returns nothing relevant, the agent answers "I couldn't find this in the company documents" — not a guess.
  • NFR-D3 — API failure handled. If the LLM call fails (rate limit, network), the agent retries once and then surfaces a user-friendly message rather than a stack trace.
  • NFR-D4 — Missing index fallback. If the vector index is missing, the app shows a clear message instructing the user to run the ingestion script — not a Python traceback.
  • NFR-D5 — No silent except: pass. Every caught exception either surfaces a user-facing message or logs a warning.

E. Security & Privacy

  • NFR-E1 — No secrets in the repo. API keys live in .env. Commit only .env.example.
  • NFR-E2 — No PII in logs. Log filenames, chunk IDs, and high-level events — not full document bodies, not API keys, not raw user questions if they contain sensitive content.
  • NFR-E3 — Local-only persistence. The vector index, the run log, and the eval report are all written locally. No outbound network calls except to the chosen LLM and embedding APIs.

F. Reproducibility & Portability

  • NFR-F1 — Five-command Quick Start. The README gets a stranger from "git clone" to a running chat in five commands or fewer: clone, install, set the API key, ingest, run.
  • NFR-F2 — OS portability. Runs on Windows, macOS, and Linux. Use pathlib.Path, not hard-coded slashes.
  • NFR-F3 — Pinned dependencies. requirements.txt uses exact version pins. Version drift can silently change retrieval or eval scores.
  • NFR-F4 — No GPU required. Both ingestion and inference run on CPU.
  • NFR-F5 — Internet required only for ingestion + LLM calls. The app does not need any other network access.

G. Maintainability & Extensibility

  • NFR-G1 — Configuration over hard-coding. Anything you might want to tune after submission (chunk size, top-k, model names, thresholds) lives in src/settings.py.
  • NFR-G2 — Adding a new tool is one file. A new tool is added by writing a Pydantic schema + function in src/tools.py and registering it — no graph rewrite.
  • NFR-G3 — Adding a domain is one config entry. Adding a fourth domain (say, Legal) should require adding the folder, updating one router-prompt entry, and re-running ingestion.

System Architecture

The system has three layers, separated cleanly so each one is easy to grade and easy to demo.

Layer 1 — Ingestion & Index
Multi-format loader (MD, PDF, DOCX) → cleaning → chunking → multilingual embedding → vector store, with domain, source_filename, and language metadata on each chunk. 		Layer 2 — LangGraph Agent
A typed graph with router, memory loader, domain-filtered retriever, agent (with three typed tools), answer composer, and memory writer. State carries the run_id and session_id through every node. 		Layer 3 — Chat UI + Observability
Chat front-end of your choice. Each turn displays the routed domain, retrieved sources, tool calls, and the cited answer. Structured JSON logs go to outputs/run_logs.jsonl; the DeepEval script reads them offline.

End-to-End Workflow

1. Receive
Employee types a question. A run_id and session_id are attached. 		2. Load Memory
Prior turns for the session are loaded into state so follow-ups resolve. 		3. Route
Router classifies into hr / it / finance, or falls back. 		4. Retrieve
Top-k chunks are fetched from the vector store, filtered by domain.
5. Decide
The agent decides whether a tool call helps (and bounded by max-step / max-tool limits). 		6. Answer
A short reply is composed in the user's language, with a "Sources" section listing filenames. 		7. Persist & Log
The turn is appended to session memory; every event is written to run_logs.jsonl. 		8. Evaluate (offline)
tests/evaluate.py runs the 10 gold cases through DeepEval and writes eval_report.json.

Optional Bonus Features

Not required for a passing grade. Pick one or two only if you finish the core requirements early and want to stand out in interviews.

  • Role-Based Access Control (RBAC): add a fake user_role (employee / manager / finance_admin) and restrict some documents to certain roles at retrieval time.
  • MCP tool server: expose your tools through an MCP server instead of in-process functions.
  • Prompt-injection guard: a simple input filter that catches obvious prompt-injection attempts in the user message.
  • Persistent cross-session memory: go beyond in-session memory and persist conversations across sessions (per user) using SQLite or a LangGraph checkpointer.
  • Custom G-Eval metric: on top of the required DeepEval metrics, define your own GEval metric (for example, "citation correctness" — does each cited file actually contain the claimed fact?) and report it in eval_report.json.
  • Hybrid retrieval: combine BM25 with vector search and report whether it improves Contextual Recall on the gold set.
  • Feedback button: a thumbs-up / thumbs-down on each answer, written to the run log for offline review.
